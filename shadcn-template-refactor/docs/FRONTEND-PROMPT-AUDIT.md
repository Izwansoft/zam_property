## 6. Response Format Inconsistencies

### 6.1 Pagination Wrapping

Backend returns two different pagination formats:

**Format A** (Listings, Vendors, etc.):
```json
{
  "data": { "items": [...], "pagination": { "page": 1, "pageSize": 20, "total": 100, "totalPages": 5 } },
  "meta": { "requestId": "..." }
}
```

**Format B** (Search, Notifications):
```json
{
  "data": [...],
  "meta": { "requestId": "...", "pagination": { "page": 1, "pageSize": 20, "totalItems": 100, "totalPages": 5 } }
}
```

**Format C** (Public Search):
```json
{
  "data": { "hits": [...], "total": 100, "aggregations": {} }
}
```

**Format D** (Admin Jobs):
```json
{
  "jobs": [...],
  "total": 8
}
```

**Part-23** §23.4 defines `PaginatedResponse<T>` using Format B only. The API client must handle all four formats.

---

### 6.2 Pagination Field Names

- Some endpoints use `total`, others use `totalItems`
- Some use `pageSize`, the public search uses `pageSize` in query but response nests differently
- Admin jobs don't use the standard wrapper at all

---

### 6.3 Search Response Shape Mismatch

**Part-25** defines `SearchResponse<T>` as:
```typescript
{ data: T[], meta: { pagination: {...}, facets?: SearchFacets } }
```

**Backend public search** returns:
```json
{ "data": { "hits": [...], "total": 100, "aggregations": {} } }
```

The `hits` vs `data[]` and `aggregations` vs `facets` naming doesn't match.
